# Internationalization

🌐 Languages: [Deutsch](../I18N.md) | [English](I18N.md)

## Principle

German is the default language of the project. English is maintained as the primary alternative language for international users. Additional languages should be possible without changing safety logic, repository modes, or public interfaces.

GitHub does not automatically translate the normal repository view based on visitor language. This repository therefore uses visible language links and concrete language-specific files.

The existing German documentation under `docs/*.md` remains the canonical default documentation. `docs/de/index.md` is the German entry point. `docs/en/index.md` is the English entry point and links to English pages or to the German canonical source where a full English page does not exist yet.

## Script Language Selection

Shared helpers live in:

- `scripts/common/i18n.ps1`
- `scripts/common/i18n.sh`

Language resolution follows this order:

1. explicit script parameter, for example `-Language en`
2. stored project configuration, for example `language: "en"` in `first-run-config.yaml`
3. `PC_AGENT_LANG`
4. German as the stable fallback

If no explicit or stored language is present, German remains the safe default.

## Product Components

Product-facing names and summaries are not duplicated in every template. They live centrally in:

- `i18n/languages.tsv`
- `i18n/product-components.tsv`

The product component catalog contains twelve directly integrated languages: `de`, `en`, `es`, `fr`, `it`, `pt`, `nl`, `pl`, `tr`, `ru`, `zh-Hans`, and `ja`. Agents can retrieve the localized product surface through `scripts/common/list-product-components.ps1` or `scripts/common/list-product-components.sh`.

Details are in [docs/en/product-components.md](product-components.md).

## UTF-8 and Unicode

All human-readable project text remains UTF-8. German prose uses real umlauts such as `ä`, `ö`, `ü`, `Ä`, `Ö`, `Ü`, and `ß`. ASCII spellings for German umlauts are avoided in German prose unless they are part of a technical token.

Template validation includes:

- PowerShell and Bash i18n tests
- product component i18n validation
- umlaut scan for German text
- PowerShell encoding check
- Bash and PowerShell syntax checks
- secret pattern scan
- `git diff --check`

## Translation Keys

Keys stay stable, small, and domain-oriented:

- `allow_baseline`
- `allow_update_maintenance`
- `first_run_missing_title`
- `snapshot_missing_error`

Technical IDs, YAML keys, status codes, file names, commands, and machine-readable protocol values are not translated.

## Adding Languages

To add another language:

1. Add the language code to `Resolve-AgentLanguage` and `agent_resolve_language`.
2. Add complete messages in `scripts/common/i18n.ps1` and `scripts/common/i18n.sh`.
3. Extend README and documentation links.
4. Add `docs/<language>/index.md`.
5. Complete `i18n/languages.tsv` and `i18n/product-components.tsv`.
6. Extend tests in `scripts/common/test-i18n.ps1`, `scripts/common/test-i18n.sh`, and `validate-product-i18n.*`.
7. Run `./scripts/common/verify-template.ps1` and `bash ./scripts/common/verify-template.sh`.

## Local Testing

```powershell
$env:PC_AGENT_LANG = 'en'
./scripts/common/test-i18n.ps1
```

```bash
PC_AGENT_LANG=en bash ./scripts/common/test-i18n.sh
```

The first-run configuration can store the language persistently in the private or local host configuration. In public `template` mode, host data still must not be written.
